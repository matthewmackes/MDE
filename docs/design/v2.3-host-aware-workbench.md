# v2.3 — Host-Aware MDE Workbench

**Locked 2026-05-22 via in-session 5-question survey + 4 adopted
best-practice suggestions.** Companion to `docs/PROJECT_WORKLIST.md
§ HA-1..HA-11`. Captures the architecture, transport choices, and
phased rollout for making the MDE Workbench manage every host on the
Mackes mesh from a single interface — not just the local machine.

## Motivation

The MDE Workbench (`crates/mde-workbench/`) is today the **default
system management interface** for Mackes Desktop Environment. It owns
nine groups (Dashboard, Look & Feel, Devices, Fleet, Network, System,
Apps, Maintain, Help) and assumes every panel reads and writes
**local** state. The existing **Fleet** group is the only group that
addresses peers, and only as a config-distribution surface
(`fleet_settings.rs:132`, `fleet_revisions.rs:103`) — operators cannot
open a peer's Network panel and toggle its Wi-Fi.

In a mesh capped at 16 peers (`v12-connectivity-scope.md` Q2) with
headless peers first-class (Q5), this gap forces operators to SSH into
each peer and run `mde` by hand. The Workbench should be **the** fleet
control surface.

## Five locked decisions

| #  | Question                                | Locked answer                                                                                                       |
|----|------------------------------------------|---------------------------------------------------------------------------------------------------------------------|
| Q1 | Transport for managing a remote peer     | **(a) SSH into peer + run `mded`.** Both reads and writes go over `ssh -o BatchMode=yes <peer> mded …`.              |
| Q2 | UX model for picking a host              | **(a) Global host switcher.** A "Viewing: <host> ▾" dropdown at the top of the sidebar flips the entire Workbench to that peer's context. Single mental model. |
| Q3 | Which groups become host-aware           | **Network, Apps, Maintain first** (Phase H3). Devices + System Phase 2 (headed peers only). Look & Feel stays local. |
| Q4 | Offline-peer UX                          | **(c) Try live, fall back to cache** from `~/QNM-Shared/<peer>/mackesd/state/`; show "offline / last seen Xm ago" indicator. Read-only fallback — writes require a live SSH session. |
| Q5 | Auth / permissions                       | **(a) Trust mesh enrollment.** Any enrolled peer can manage any other. No per-remote AdminSession prompt. Local `host:self` retains today's pkexec path. |

## Adopted refinements (S1–S4)

* **S1 — Phased rollout, not big-bang.** Only Network / Apps /
  Maintain become host-aware in v2.3. Devices / System / Look & Feel
  stay local.
* **S2 — Local is just `host:self`.** Eliminate the "is this local or
  remote?" code fork. Today's local-only panels become the trivial
  case of the remote code path. The global host switcher always lists
  `host:self` as the first entry.
* **S3 — Heartbeat-published state cache.** Every peer's `mackesd`
  publishes a per-key state snapshot to its own
  `~/QNM-Shared/<peer>/mackesd/state/<key>.json` on every heartbeat
  tick. The Workbench reads cache when a peer is unreachable. This is
  the offline-read-source-of-truth.
* **S4 — Append-only fleet audit log.** Every cross-host action is
  recorded at `~/QNM-Shared/.mde-audit/<YYYY-MM-DD>.jsonl` with a
  SHA-256 hash chain. **Especially load-bearing under Q5=(a) flat
  mesh trust:** with no per-peer auth gate, the audit log is the
  primary accountability mechanism.

**S5 (stage-then-apply changeset) dropped.** Without offline-write
queueing (Q1=a means writes are live SSH) or multi-host fan-out
(Q2=a focuses on one host at a time), the changeset model loses its
load-bearing role. Writes apply immediately.

## Honoring the "no networked API" design lock

The v12.0 mesh control plane locks
(`v12.0-enterprise-mesh.md:41-43, 137-152`):

> No networked API exists — peer communication happens through the
> shared filesystem only.

SSH-into-peer + run-mded **does not break this lock**:

* SSH is the existing transport for inter-peer access, documented
  at `mackes/mesh_ssh.py:1-16`. We add no new sockets, daemons, or
  firewall holes.
* The state cache (S3) is a filesystem read like every other
  inter-peer state read.

The only daemon-side change is `mackesd` publishing per-key state
snapshots to its own peer directory on every heartbeat — an
extension of the heartbeat pattern already running.

## Architecture

### Layer additions to `crates/mde-workbench/`

```text
┌────────────────────────────────────────────────────────────────┐
│  UI: src/sidebar.rs — top-bar HostSwitcher widget               │
│    Dropdown listing host:self + every enrolled peer             │
├────────────────────────────────────────────────────────────────┤
│  UI: src/panels/<group>/<panel>.rs                              │
│    Host-aware panels read app.current_host(); show "local only" │
│    banner when current_host != SELF and panel is not host-aware │
├────────────────────────────────────────────────────────────────┤
│  Domain: src/host.rs                                            │
│    HostId, HostInfo, HostRegistry — heartbeat-watching loader   │
├────────────────────────────────────────────────────────────────┤
│  Service: src/backend.rs (HostBackend trait)                    │
│    .get(host, key)  → live read; falls back to cache when down  │
│    .set(host, key)  → live SSH write; fails when host is down   │
│    .cached(host, key) → ~/QNM-Shared/<peer>/mackesd/state/<key> │
├────────────────────────────────────────────────────────────────┤
│  Transport: src/transport/ssh.rs                                │
│    ssh -o BatchMode=yes -o ConnectTimeout=3 <peer> mded <args>  │
├────────────────────────────────────────────────────────────────┤
│  Audit: src/audit.rs — appends ~/QNM-Shared/.mde-audit/*.jsonl  │
└────────────────────────────────────────────────────────────────┘
```

### `HostBackend` trait (Rust sketch)

```rust
#[async_trait]
pub trait HostBackend: Send + Sync {
    async fn get(&self, host: HostId, key: &str) -> Result<Value, HostError>;
    async fn set(&self, host: HostId, key: &str, val: Value) -> Result<RevisionId, HostError>;
    fn cached(&self, host: HostId, key: &str) -> Option<Value>;
}
```

Two implementations dispatched by `HostId`:

* `LocalHostBackend` — today's DBus path. Active only for
  `host == HostId::SELF`.
* `RemoteHostBackend` — `get` and `set` both run over
  `transport::ssh`. `set` returns `HostError::Unreachable` if the
  peer is offline; `get` falls through to `cached(host, key)` on
  unreachable and surfaces an "offline" indicator to the panel.

A composite `RoutingHostBackend` dispatches by `HostId` so callers
never branch on local-vs-remote.

### Global HostSwitcher

A single state slot in the Workbench app: `current_host: HostId`,
default `HostId::SELF`. The sidebar header renders a dropdown:

```text
╔════════════════════════════════╗
║  Viewing: host-self ▾          ║
║  ────────────────────────────  ║
║  • host-self  (this machine)   ║
║  • lab-bench-01    ● online    ║
║  • lab-bench-02    ● offline   ║
║  • kitchen-pi      ● online    ║
╚════════════════════════════════╝
```

Selecting an entry updates `current_host` and re-renders the active
panel. Host-aware panels (Network, Apps, Maintain) read
`current_host` and parameterize their `HostBackend` calls. Non-aware
panels (Dashboard, Look & Feel, Devices, System) render a top
banner when `current_host != SELF`:

> *"This panel only edits the local machine. Switch back to
> host-self to make changes."*

Default behavior on switcher change: do not auto-navigate. If the
operator is viewing a non-host-aware panel and switches hosts, the
banner appears in place. If they switch back to `host:self` the
banner disappears.

### State cache (S3)

```text
~/QNM-Shared/<peer>/mackesd/
├── heartbeat.json              # liveness (existing)
└── state/
    └── <key>.json              # NEW — per-key state snapshots
```

`mackesd` writes one snapshot per host-aware key on every heartbeat
tick (currently 10s). Initial keys covered in v2.3:

* `network.wifi.*` (SSID list, active SSID, signal)
* `network.firewall.*` (enabled, open ports)
* `network.mesh.*` (peer reachability, last sync)
* `apps.installed` (package list with versions)
* `maintain.snapshots.*` (snapshot manifest list)
* `maintain.updates.*` (pending updates summary)
* `maintain.health.*` (cpu, memory, disk, services)

The Workbench reads cache only when an SSH read fails with
`Unreachable`. Cache is **never** the primary read source for an
online peer — operators need to trust that what they see is live.

### Audit log (S4)

Every cross-host action appends to
`~/QNM-Shared/.mde-audit/<YYYY-MM-DD>.jsonl`:

```json
{
  "ts": "2026-05-22T14:32:14Z",
  "actor_host": "host-A",
  "actor_user": "operator",
  "target_host": "host-B",
  "action": "set",
  "key": "network.firewall.enabled",
  "before": false,
  "after": true,
  "result": "applied",
  "prev_hash": "sha256:…",
  "this_hash": "sha256:…"
}
```

SHA-256 hash chain mirrors `crates/mackesd/src/audit.rs::next_hash`.
Tampering with any entry breaks chain verification.

**Local-host actions are also audited** (with
`target_host == actor_host`) — gives a uniform log regardless of
whether the operator was managing local or remote.

The Workbench includes a "View audit log for this host" link in the
HostSwitcher dropdown, filtering entries by `target_host`.

### Failure modes (Q4 detail)

| Scenario                                | UI behavior                                                                                       |
|------------------------------------------|---------------------------------------------------------------------------------------------------|
| Peer online, read succeeds              | Live data, no badge.                                                                              |
| Peer online, read fails (non-network)   | Error banner with diagnostic; cache shown as read-only fallback if available.                     |
| Peer offline, cache available           | Cached data + amber badge: "Offline — last seen 3m ago. Changes disabled."                        |
| Peer offline, no cache available        | Empty state: "Host is offline and has no cached state. Bring the host online to view settings."   |
| Peer online, write fails (auth/protocol) | Toast with diagnostic; the panel stays editable so the operator can retry.                        |
| Peer offline, write attempted           | Refused at the HostBackend layer — the apply button is disabled when `is_online == false`; no inbox queue. |

## Phased rollout

Each phase ships independently. Worklist tasks `HA-1..HA-11` in
`docs/PROJECT_WORKLIST.md § HA-1..HA-11` track the units of work.

### Phase H1 — Foundations (no UI changes)

| Task  | Scope                                                                   |
|-------|--------------------------------------------------------------------------|
| HA-1  | `HostId`, `HostInfo`, `HostRegistry` in `crates/mde-workbench/src/host.rs`. Watches `~/QNM-Shared/*/mackesd/heartbeat.json` for liveness. |
| HA-2  | `HostBackend` trait + `LocalHostBackend` + `RoutingHostBackend` shell.   |
| HA-3  | State-cache protocol — `mackesd` publishes `state/<key>.json` per heartbeat tick. |
| HA-4  | Audit log helper + SHA-256 hash chain. Local-host actions also logged.   |

### Phase H2 — SSH transport

| Task  | Scope                                                                   |
|-------|--------------------------------------------------------------------------|
| HA-5  | `transport::ssh` — wraps `ssh -o BatchMode=yes -o ConnectTimeout=3 <peer> mded …`. Errors: `Unreachable`, `AuthRefused`, `Timeout`. |
| HA-6  | `RemoteHostBackend` implementation — `get`/`set` via SSH, `cached` via filesystem; `get` falls back to cache on `Unreachable`. |

### Phase H3 — UI

| Task  | Scope                                                                   |
|-------|--------------------------------------------------------------------------|
| HA-7  | Global `HostSwitcher` widget at top of sidebar. App state `current_host: HostId`. Dropdown lists `host:self` first, then every known peer. Non-host-aware panels render the "local only" banner when `current_host != SELF`. |
| HA-8  | Refactor Network panels (Wi-Fi, mesh, VPN, firewall) — read `current_host`, call `HostBackend::get/set(current_host, key)`. Offline empty-state. |
| HA-9  | Refactor Apps installer to be host-aware. install/remove routes to peer's mded via SSH. |
| HA-10 | Refactor Maintain panels (Snapshots, Health, Updates) to be host-aware. |

### Phase H4 — Polish + docs

| Task  | Scope                                                                   |
|-------|--------------------------------------------------------------------------|
| HA-11 | Offline empty-state polish, help docs (`docs/help/hosts.md`, `docs/help/host-aware-panels.md`), screenshot pair (local vs. host-scoped), CHANGELOG entry. |

## Out of scope for v2.3

* Devices, System, Look & Feel becoming host-aware (Phase 2,
  post-v2.3 — open a new design doc when scheduled).
* Cross-host search/filter ("show me every peer with firewall off") —
  natural follow-up after HA-10 ships.
* Multi-select / "act on N hosts at once" UI — the Fleet group already
  handles fleet-wide pushes via `fleet_settings.rs:132`. Adding
  multi-select to the global host switcher is a separate decision.
* Offline-write queueing — Q1=(a) pure-SSH explicitly rules this out.
  Operators see "host offline, changes disabled" rather than a queued
  write that might apply later.
* Per-remote `AdminSession` — Q5=(a) flat mesh trust explicitly rules
  this out. If a future audit reveals abuse, revisit by re-voting on
  Q5 in a Phase 2 design doc.
* Mesh-internal RPC daemon — Q1=(a) explicitly rules this out. Would
  break the v12.0 "no networked API" lock.
